# Tiny Second-hand Shopping Platform 개발 보고서

- 과제명: Tiny Second-hand Shopping Platform (중고나라 미니)
- GitHub: https://github.com/stacknoah/oneandaonly_secondhand
- 스택: Python 3.13, Flask, Flask-SocketIO, SQLite, Jinja2
- 개발 방식: Claude Code(AI)를 활용한 전 과정 진행 (요구사항 분석 → 설계 → 구현 → 테스팅 → 유지보수 점검)

## 1. 요구사항 분석

### 1.1 기능 요구사항

강의 자료의 최소 요구사항을 기능 단위로 분해

| 영역 | 요구사항 | 세부 기능 |
|---|---|---|
| 유저 관리 | 플랫폼 가입 가능 | 회원가입, 로그인, 사용자 프로필 조회, 마이페이지(소개글·비밀번호 변경), 아이디 중복 금지, DB 관리 |
| 상품 관리 | 상품 등록·조회 가능 | 상품 등록, 내 상품 관리, 상품명·가격·사진 표시, 전체 공개 조회, 목록은 이름만 → 클릭 시 상세 페이지 |
| 유저 소통 | 사용자 간 소통 가능 | 실시간 전체 채팅, 1:1 채팅 |
| 악성 차단 | 악성 유저·상품 차단 | 신고 기능(사유 작성), 누적 신고 시 상품 차단, 누적 신고 시 유저 휴면 전환 |
| 송금 | 유저 간 송금 가능 | 잔액 기반 송금, 거래 내역 |
| 검색 | 상품 검색 가능 | 상품명 키워드 검색 |
| 관리자 | 모든 요소 관리 가능 | 유저·상품·신고 전체 조회 및 제재 |

### 1.2 비기능 요구사항

- 보안: 과제 핵심 요구사항, "최대한 보안 약점이 없도록 할 것" → 6장에서 상세
- 디자인: 실사용 가능한 수준의 일관된 UI (단일 CSS)
- 서버 속도: SQLite 인덱스(판매자·신고 대상·수신자), 채팅 히스토리 LIMIT 조회

### 1.3 요구사항 변경·추가 사항

강의 자료에서 논리적으로 타당한 범위의 변경·추가 허용 → 다음을 추가

- 신고 기준치: "일정 횟수"를 3회로 확정 (설정값 `REPORT_THRESHOLD`로 분리, 변경 용이)
- 송금을 위해 유저에게 초기 잔액 100,000원 지급 (외부 결제 연동이 없는 과제 환경에서 송금 기능을 검증 가능하게 하는 최소 장치)
- 관리자 계정은 신고 누적 자동 휴면 대상에서 제외 (신고 기능을 악용한 운영 계정 잠금 공격 방지)

## 2. 시스템 분석 (송금·검색·관리자)

과제 요구에 따라 이 세 기능은 시스템 분석부터 진행

### 2.1 송금

- 행위자: 로그인 유저(송금인), 수신 유저
- 흐름: 수신자 아이디·금액 입력 → 서버 검증(정수, 1~10,000,000원, 수신자 존재·활성, 자기 자신 금지) → 잔액 확인 → 차감·가산·기록을 하나의 트랜잭션으로 처리 → 거래 내역 표시
- 핵심 위험: 금액 파라미터 조작(음수·문자열), 동시 요청에 의한 이중 지불, 잔액 검증 우회
- 결정: DB 레벨 CHECK(amount > 0, balance >= 0) + 애플리케이션 레벨 검증 + `BEGIN IMMEDIATE` 트랜잭션의 3중 방어

### 2.2 검색

- 행위자: 모든 방문자 (비로그인 포함, 상품은 전체 공개이므로)
- 흐름: 검색어 입력 → 활성 상품 중 상품명 LIKE 매칭 → 목록 표시
- 핵심 위험: 검색어를 통한 SQL Injection
- 결정: LIKE 패턴도 파라미터 바인딩으로 처리, 차단된 상품은 검색 결과에서 제외

### 2.3 관리자

- 행위자: role이 admin인 유저
- 흐름: 대시보드(통계) → 유저 관리(휴면/활성/삭제) → 상품 관리(차단/해제/삭제) → 신고 내역 조회
- 핵심 위험: 권한 검증 누락 시 일반 유저의 관리 기능 접근(권한 상승), 관리자 자신을 제재하는 실수
- 결정: 모든 관리자 라우트에 `admin_required` 데코레이터 일괄 적용, 자기 자신 상태 변경·삭제 금지, 관리자 계정 시드는 init-db에서 생성

## 3. 시스템 설계

### 3.1 아키텍처

```
브라우저 ── HTTP ──> Flask (Blueprint: auth / products / chat / reports / wallet / admin)
        ── WebSocket ──> Flask-SocketIO (전체 채팅, 1:1 채팅)
                              │
                           SQLite (users, products, reports, messages, transfers)
```

- 요청 공통 처리: `before_request`에서 세션 유저 로드(휴면·삭제 계정 즉시 무효화) + 전 POST CSRF 검증
- 소켓 이벤트: 매 이벤트마다 세션 기준 인증·상태 재확인, 클라이언트 제공 값 불신

### 3.2 웹페이지 설계 (강의 자료 명세 페이지 전부 포함)

| 명세 페이지 | 구현 경로 |
|---|---|
| 기본 페이지 | `/` |
| 회원 가입 페이지 | `/register` |
| 로그인 페이지 | `/login` |
| 전체 상품 조회 및 채팅 페이지 | `/products/` (상품 목록 + 전체 채팅 사이드바) |
| 새 상품 등록 페이지 | `/products/new` |
| 상품 상세 조회 페이지 | `/products/<id>` |
| 신고 페이지 | `/report/new?type=&id=` |
| (추가) 프로필 / 마이페이지 | `/users/<id>`, `/mypage` |
| (추가) 1:1 채팅 | `/chat/dm`, `/chat/dm/<peer_id>` |
| (추가) 지갑·송금 | `/wallet/` |
| (추가) 관리자 | `/admin/` 이하 |

### 3.3 데이터베이스 설계

명세 3테이블(사용자·상품·신고)에 채팅·송금 저장을 위해 2테이블 추가

| 테이블 | 컬럼 | 비고 |
|---|---|---|
| users | id, username(UNIQUE), password_hash, bio, role, status, balance, created_at | 아이디 중복은 UNIQUE 제약으로 차단, 비밀번호는 해시만 저장 |
| products | id, title, description, price, image, seller_id(FK), status, created_at | status로 차단 상태 관리, CHECK(price >= 0) |
| reports | id, reporter_id(FK), target_type, target_id, reason, created_at | UNIQUE(reporter, target)로 중복 신고 차단 |
| messages | id, sender_id(FK), receiver_id(FK, NULL=전체 채팅), content, created_at | 전체 채팅과 DM을 한 테이블로 관리 |
| transfers | id, sender_id(FK), receiver_id(FK), amount, created_at | CHECK(amount > 0) |

## 4. 시스템 구현

- 구조: 앱 팩토리(`app/__init__.py`) + 기능별 Blueprint 6개, 공통 보안 로직은 `app/security.py`로 분리
- DB 접근: sqlite3 표준 라이브러리, 요청 스코프 커넥션(g), 외래키 강제(`PRAGMA foreign_keys = ON`)
- 실시간 채팅: Flask-SocketIO, 전체 채팅은 lobby 룸, 1:1은 `dm:<작은id>:<큰id>` 룸을 서버가 세션 신원으로 계산
- 파일 업로드: 확장자 화이트리스트 → UUID 파일명으로 저장 → 정적 경로 서빙
- 관리 CLI: `flask init-db`(스키마 + 관리자 시드), `flask seed-demo`(데모 데이터)
- 실행·설치 방법은 README.md에 명시

## 5. 개발 과정에서 확인한 보안 약점과 조치

구현 중 자체 점검(6장 체크리스트 기반)에서 확인·보완한 약점. 취약점을 개별 버그가 아니라 클래스 단위로 관리

| # | 약점 클래스 | 발견 지점 | 조치 내용 |
|---|---|---|---|
| 1 | SQL Injection | 검색어·로그인 등 유저 입력이 쿼리에 유입되는 전 지점 | 문자열 포매팅 전면 금지, 전 쿼리 `?` 파라미터 바인딩 (검색 LIKE 패턴 포함) |
| 2 | Stored XSS | 상품명·설명·소개글·채팅 메시지 | Jinja autoescape 유지(우회 없음), 채팅 동적 렌더링은 innerHTML 대신 textContent 사용 |
| 3 | CSRF | 상품 삭제·송금·신고 등 상태 변경 POST | 세션 발급 CSRF 토큰을 모든 폼에 삽입, `before_request`에서 전 POST 일괄 검증 (compare_digest로 비교) |
| 4 | 비밀번호 평문 저장 | users 테이블 | werkzeug scrypt 해시 저장, 검증은 `check_password_hash` |
| 5 | IDOR (수평 권한 상승) | `/products/<id>/edit`, `/delete` | 소유자 또는 관리자만 허용하는 `get_owned()` 공통 검증, 실패 시 403 |
| 6 | 수직 권한 상승 | `/admin/*` | `admin_required` 데코레이터 일괄 적용, 비로그인·일반 유저 403 |
| 7 | 악성 파일 업로드 | 상품 이미지 | 확장자 화이트리스트(png/jpg/jpeg/gif/webp), 원본 파일명 폐기 후 UUID 재명명(경로 조작·덮어쓰기 차단), 2MB 크기 제한 |
| 8 | 송금 금액 조작·이중 지불 | `/wallet/transfer` | 서버측 정수·범위 검증(음수·0·한도 초과 거부), 자기 자신 송금 거부, `BEGIN IMMEDIATE` 트랜잭션으로 잔액 확인·차감·기록의 원자성 보장, DB CHECK 제약으로 최종 방어 |
| 9 | 로그인 브루트포스 | `/login` | 계정별 5회 실패 시 5분 잠금, 아이디 존재 여부 비노출(계정 열거 방지) |
| 10 | 세션 관련 약점 | 쿠키·휴면 계정 | HttpOnly + SameSite=Lax, 로그인 시 세션 재발급(고정 공격 방지), 휴면·삭제 계정은 요청 시점에 세션 즉시 무효화 |
| 11 | 소켓 인증 우회 | SocketIO 이벤트 | 이벤트마다 세션 기준 인증·활성 상태 재확인, 방 이름은 서버가 계산(임의 방 조인 차단), 미인증 소켓 메시지 폐기 |
| 12 | 채팅 도배 (DoS성 남용) | 전체·1:1 채팅 | 유저별 3초 5건 rate limit, 메시지 500자 제한 |
| 13 | 신고 기능 남용 | `/report/new` | 동일 대상 중복 신고 UNIQUE 차단, 자기 자신·본인 상품 신고 금지, 관리자 계정 자동 휴면 제외 |
| 14 | 정보 노출 | 에러 처리 | debug 비활성 실행, 일반화된 400/403/404/413 에러 페이지, 스택트레이스 비노출 |
| 15 | 시크릿 하드코딩 | SECRET_KEY·관리자 비밀번호 | 환경변수 우선(SECRET_KEY, ADMIN_PASSWORD), 데모 기본값은 README에 한계 명시 |
| 16 | 비밀번호 변경 탈취 악용 | 마이페이지 | 변경 시 현재 비밀번호 재확인 필수 (세션 탈취만으로 계정 탈취 불가) |

PASS  일반 유저 관리자 페이지 차단

## 7. 유지보수

실사용 관점 점검에서 확인한 한계와 개선 방향

- 로그인 잠금·채팅 rate limit이 인메모리 → 서버 재시작 시 초기화, 다중 프로세스 환경에서 미공유. 운영 전환 시 Redis 등 외부 스토어로 이전 필요
- SQLite는 동시 쓰기에 약함 → 사용자 증가 시 PostgreSQL 전환 (스키마는 표준 SQL로 작성해 이전 비용 최소화)
- HTTPS 미적용 (로컬 과제 환경) → 배포 시 TLS 종단(리버스 프록시) 및 Secure 쿠키 플래그, WSS 적용
- 이미지가 확장자 기반 검증 → 매직 바이트 검증, 이미지 재인코딩 추가 여지
- 채팅 기록 무한 증가 → 보존 기간 정책 및 페이지네이션 필요
- 유저 삭제가 하드 삭제(CASCADE) → 운영에서는 소프트 삭제 + 감사 로그가 적합
