- 제출일: 2026. 7. 23
- GitHub: https://github.com/stacknoah/oneandaonly_secondhand
- 개발 환경: Python 3.13, Flask, Flask-SocketIO, SQLite

## 1. 개요

중고거래가 가능한 웹 플랫폼을 개발하는 과제로, 요구사항 분석부터 설계, 구현, 테스팅, 유지보수 검토까지 소프트웨어 개발 주기 전체를 진행하였다. 과제 안내에 따라 개발 전 과정에서 Claude Code를 적극 활용하였고, 요구사항 확정과 보안 설계 방향 결정, 최종 검증은 직접 수행하였다.

기술 스택은 Flask와 SQLite를 선택하였다. 과제 규모에서 프레임워크 오버헤드가 가장 적고, 실시간 채팅 요구사항은 Flask-SocketIO의 WebSocket으로 해결할 수 있기 때문이다.

- 외부 결제 연동이 없는 환경에서 송금을 검증할 수 있도록 가입 시 초기 잔액 100,000원 지급
- 신고 기능으로 관리자 계정을 잠글 수 없도록 관리자는 자동 휴면 대상에서 제외

## 3. 시스템 분석

과제 요구에 따라 송금, 검색, 관리자 기능은 시스템 분석을 먼저 진행하였다.

### 3.1 송금

로그인한 유저가 수신자 아이디와 금액을 입력하면 서버가 검증 후 잔액을 이동한다. 이 기능에서 가장 위험한 지점은 금액 파라미터 조작과 동시 요청이다. 음수 금액을 보내 잔액을 역으로 늘리거나, 같은 요청을 동시에 보내 잔액 검증을 통과한 뒤 이중으로 출금시키는 공격이 가능하다. 따라서 금액은 서버에서 정수와 범위를 검증하고, 잔액 확인부터 차감, 가산, 기록까지 하나의 트랜잭션으로 묶고, 데이터베이스에도 잔액과 금액에 CHECK 제약을 걸어 세 겹으로 방어하기로 결정하였다.

### 3.2 검색

상품은 전체 공개이므로 검색도 비로그인 상태에서 가능해야 한다. 검색어가 SQL 쿼리에 직접 들어가면 SQL Injection이 성립하므로 LIKE 패턴을 포함한 모든 쿼리를 파라미터 바인딩으로 처리한다. 차단된 상품은 검색 결과에서 제외한다.

### 3.3 관리자

관리자는 유저의 휴면 전환과 삭제, 상품의 차단과 삭제, 신고 내역, 채팅 메시지, 송금 내역까지 플랫폼의 모든 데이터를 조회하고 제재할 수 있어야 한다. 권한 검증이 한 곳이라도 빠지면 일반 유저가 관리 기능에 접근하는 권한 상승으로 이어지므로, 관리자 라우트 전체에 공통 데코레이터를 적용한다. 관리자가 실수로 자기 자신을 휴면이나 삭제 처리하는 일은 서버에서 막는다.

## 4. 시스템 설계

### 4.1 아키텍처

```
브라우저 ─ HTTP ─────  Flask (auth, products, chat, reports, wallet, admin)
        ─ WebSocket ─  Flask-SocketIO (전체 채팅, 1대1 채팅)
                            │
                         SQLite
```

모든 요청은 공통 전처리를 거친다. 세션의 유저를 조회하면서 휴면이나 삭제된 계정이면 세션을 즉시 무효화하고, POST 요청은 전부 CSRF 토큰을 검증한다. 소켓 이벤트는 매번 세션 기준으로 인증 상태를 다시 확인하며 클라이언트가 보낸 값을 신뢰하지 않는다.

### 4.2 페이지 설계

강의 자료에 명세된 페이지를 모두 포함하고 필요한 페이지를 추가하였다.

| 구분 | 페이지 | 경로 |
|---|---|---|
| 명세 | 기본 페이지 | / |
| 명세 | 회원 가입 페이지 | /register |
| 명세 | 로그인 페이지 | /login |
| 명세 | 전체 상품 조회 및 채팅 페이지 | /products/ |
| 명세 | 새 상품 등록 페이지 | /products/new |
| 명세 | 상품 상세 조회 페이지 | /products/[id] |
| 명세 | 신고 페이지 | /report/new |
| 추가 | 프로필, 마이페이지 | /users/[id], /mypage |
| 추가 | 내 상품 관리 | /products/mine |
| 추가 | 1대1 채팅 목록과 대화 | /chat/dm, /chat/dm/[id] |
| 추가 | 지갑과 송금 | /wallet/ |
| 추가 | 관리자 | /admin 이하 6개 페이지 |

### 4.3 데이터베이스 설계

명세된 사용자, 상품, 신고 테이블에 채팅과 송금 저장용 테이블을 추가하였다.

| 테이블 | 컬럼 | 설계 의도 |
|---|---|---|
| users | id, username, password_hash, bio, role, status, balance, created_at | 아이디 중복은 UNIQUE 제약으로 차단, 비밀번호는 해시만 저장, status로 휴면 관리 |
| products | id, title, description, price, image, seller_id, status, created_at | status로 차단 상태 관리, 가격에 CHECK 제약 |
| reports | id, reporter_id, target_type, target_id, reason, created_at | 신고자와 대상 조합에 UNIQUE 제약을 걸어 중복 신고 차단 |
| messages | id, sender_id, receiver_id, content, created_at | receiver_id가 없으면 전체 채팅, 있으면 1대1 채팅 |
| transfers | id, sender_id, receiver_id, amount, created_at | 금액에 양수 CHECK 제약 |

## 5. 구현

앱 팩토리 구조에 기능별 블루프린트 6개로 나누어 구현하였다. 공통 보안 로직은 별도 모듈로 분리하였다.

```
run.py                  실행 진입점
app/
  __init__.py           앱 팩토리, 공통 전처리, 에러 핸들러
  db.py                 SQLite 연결 관리, init-db, seed-demo 명령
  schema.sql            테이블 스키마
  security.py           CSRF, 인증 데코레이터, 입력 검증, 로그인 잠금
  routes/               auth, products, chat, reports, wallet, admin
  templates/, static/   Jinja 템플릿, CSS, 업로드 저장소
```

구현에서 중요했던 결정 사항은 다음과 같다.

- 1대1 채팅방 이름은 두 유저 id를 정렬해 서버가 계산하며, 클라이언트가 방 이름을 지정할 수 없게 하여 임의 방 참여를 차단
- 업로드 이미지는 확장자 화이트리스트 검사 후 원본 파일명을 버리고 UUID로 저장
- 유저를 조회하는 공통 전처리에서 계정 상태를 함께 확인하므로, 휴면 전환된 유저는 다음 요청부터 바로 로그아웃 처리
- 설치와 실행 방법은 저장소 README.md에 명시

## 6. 개발 과정에서 확인한 보안 약점과 조치

구현하면서 기능마다 공격 가능한 지점을 점검하고 조치하였다. 개별 버그가 아니라 취약점 클래스 단위로 정리하였다.

| 약점 | 위험한 지점 | 조치 |
|---|---|---|
| SQL Injection | 검색어, 로그인 등 유저 입력이 쿼리에 들어가는 모든 지점 | 문자열 조합 대신 전 쿼리 파라미터 바인딩, LIKE 패턴 포함 |
| Stored XSS | 상품명, 설명, 소개글, 채팅 메시지 | Jinja 자동 이스케이프 유지, 채팅의 동적 렌더링은 innerHTML 대신 textContent 사용 |
| CSRF | 상품 삭제, 송금, 신고 등 상태를 바꾸는 모든 POST | 세션에 토큰을 발급해 모든 폼에 삽입하고 공통 전처리에서 검증 |
| 비밀번호 평문 저장 | users 테이블 | scrypt 해시로 저장하고 원문은 어디에도 남기지 않음 |
| 수평 권한 상승 | 상품 수정과 삭제 | 소유자이거나 관리자인지 서버에서 확인, 실패 시 403 |
| 수직 권한 상승 | 관리자 기능 전체 | admin_required 데코레이터를 모든 관리자 라우트에 적용 |
| 악성 파일 업로드 | 상품 이미지 | 이미지 확장자만 허용, 파일명은 UUID로 교체, 크기 2MB 제한 |
| 송금 금액 조작과 이중 지불 | 송금 요청 | 서버에서 정수와 범위 검증, 자기 자신에게 송금 금지, 잔액 확인부터 기록까지 단일 트랜잭션, DB CHECK 제약 |
| 로그인 무차별 대입 | 로그인 | 계정별 5회 실패 시 5분 잠금, 아이디 존재 여부는 응답에서 구분되지 않게 처리 |
| 세션 탈취와 고정 | 세션 쿠키 | HttpOnly와 SameSite 속성 적용, 로그인 성공 시 세션 재발급 |
| 휴면 계정의 기존 세션 | 로그인 이후 상태 변화 | 매 요청 계정 상태를 확인해 휴면 전환 즉시 세션 무효화 |
| 소켓 인증 우회 | 채팅 이벤트 | 이벤트마다 세션 기준 재인증, 미인증 소켓의 메시지는 폐기 |
| 채팅 도배 | 전체 채팅과 1대1 채팅 | 유저별 3초에 5건 제한, 메시지 길이 500자 제한 |
| 신고 기능 남용 | 신고 접수 | 동일 대상 중복 신고 차단, 자기 자신과 본인 상품 신고 금지 |
| 내부 정보 노출 | 에러 응답 | debug 모드 없이 실행, 일반화된 에러 페이지로 통일 |
| 시크릿 하드코딩 | 비밀 키, 관리자 초기 비밀번호 | 환경변수로 주입받는 구조, 데모 기본값의 한계는 README에 명시 |
| 비밀번호 변경 악용 | 마이페이지 | 변경 시 현재 비밀번호를 다시 확인해 세션 탈취만으로는 변경 불가 |

### 7.3 자동화 테스트

수동 확인만으로는 회귀를 놓칠 수 있어 tests/smoke_test.py로 자동화하였다. 서버를 띄운 상태에서 실제 HTTP 요청을 보내 기능과 보안 20개 항목을 점검하며, 전 항목 통과를 확인하였다.

```
PASS  회원가입, 아이디 중복 거부, 아이디 형식 검증
PASS  CSRF 토큰 없는 요청 거부
PASS  상품 등록, 목록 노출, 검색, SQL Injection 시도 무해화, 음수 가격 거부
PASS  타인 상품 삭제와 수정 차단
PASS  상품명 XSS 이스케이프
PASS  신고 3회 누적 시 상품 자동 차단, 동일 대상 중복 신고 차단
PASS  송금 정상 처리, 한도 초과와 음수와 자기 자신 송금 거부
PASS  일반 유저의 관리자 페이지 차단
PASS  로그인 5회 실패 시 잠금
20/20 통과
```

WebSocket은 별도 클라이언트 스크립트로 전체 채팅 송수신과 비로그인 소켓 차단을 확인하였고, 유저 신고 누적 시 휴면 전환과 기존 세션 무효화, 마이페이지 수정, 비로그인 상품 조회도 개별 검증하였다.

## 8. 유지보수

실사용 관점에서 점검한 한계와 개선 방향이다.

- 로그인 잠금과 채팅 전송 제한이 메모리에 저장되어 서버 재시작 시 초기화됨. 운영 환경에서는 Redis 같은 외부 저장소로 이전 필요
- SQLite는 동시 쓰기에 취약하므로 사용자가 늘면 PostgreSQL 전환 필요. 표준 SQL로 작성해 이전 비용은 낮음
- 로컬 과제 환경이라 HTTPS 미적용. 배포 시 TLS와 Secure 쿠키, WSS 적용 필요
- 이미지 검증이 확장자 기반이므로 매직 바이트 검사와 재인코딩을 추가할 여지
- 유저 삭제가 연관 데이터까지 지우는 하드 삭제 방식. 운영에서는 소프트 삭제와 감사 로그가 적합
- 채팅 기록이 계속 쌓이므로 보존 기간 정책 필요
